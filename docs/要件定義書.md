# Walleca 要件定義書

## 1. 研究背景・目的

### 1.1 背景
- 動画配信、音楽配信、クラウドサービスなどのサブスクリプションが増え、個人の支出は固定費と変動費の両面で管理が必要になっている。
- 既存の家計簿アプリは入力負荷が高く継続しづらく、サブスク管理アプリは日常支出まで扱えないことが多い。

### 1.2 目的
- レシート OCR による支出登録
- サブスクリプション管理
- 一覧、カレンダー、集計を中心とした柔軟な閲覧

上記を統合した、Web 優先の個人支出管理システムを設計・実装する。研究上は API 中心のクライアント分離構成を採用し、保守性と拡張性を重視する。

## 2. システム概要

### 2.1 システム名
- Walleca

### 2.2 対象ユーザー
- 個人ユーザー
- 日本国内利用を主に想定
- 共有家計、グループ家計は対象外

### 2.3 提供形態
- 初期段階は Web アプリを優先
- 将来的にモバイルアプリへ展開可能な構成とする
- 自宅サーバー上での運用を前提とする

## 3. 基本方針

### 3.1 アーキテクチャ方針
- Web クライアントは Next.js App Router を採用する
- クライアントは表示と入力に専念し、業務ロジックは API サーバーに集約する
- API を中心に設計し、将来のモバイル対応を容易にする

### 3.2 OCR 方針
- OCR エンジンには Tesseract を利用する
- OCR は非同期ジョブとして処理する
- アップロード画像は永続保存しない
- OCR 結果はユーザー確認を前提とし、未確定データは破棄可能とする

### 3.3 MVP 方針
- 卒業研究として完成させることを優先する
- MVP では個人利用に必要な最小機能を確実に実装する
- 商用運用や大規模スケール対応は対象外とする

## 4. 機能要件

### 4.1 認証・ユーザー管理
- ユーザー登録
- ログイン、ログアウト
- メールアドレスとパスワードによる認証
- Web クライアント間で同一アカウントのデータを同期

補足:
- パスワード再発行、メール認証は MVP 対象外
- 将来的な追加機能として扱う

### 4.2 支出管理

#### 4.2.1 取引登録
- 手動で取引を登録できる
- 画像なしでも支出、収入、調整を登録できる
- レシート OCR 確定後に取引を生成できる

#### 4.2.2 取引項目
- 種別: 支出、収入、調整
- 金額
- 日付
- 店名
- カテゴリ
- 支払い方法
- メモ

#### 4.2.3 カテゴリ
- ユーザーごとのカテゴリマスタを持つ
- 未設定カテゴリを許容する
- 初期カテゴリを用意する
- ユーザーが追加、編集できる

#### 4.2.4 支払い方法
- 固定候補から選択する
- 候補は現金、クレジットカード、デビットカード、電子マネー、QR コード決済、銀行振込、その他とする

### 4.3 レシート OCR

#### 4.3.1 入力
- Web から画像をアップロードできる
- 対応形式は JPEG、PNG
- ファイルサイズ制限を設ける

#### 4.3.2 抽出項目
- 店名
- 購入日
- 合計金額
- OCR 全文テキスト

#### 4.3.3 確認フロー
- OCR 結果はユーザーが確認、修正してから確定する
- 元画像は確認画面では保持しない
- OCR 全文テキストは確認前のみ保持し、確定後に削除する
- 画像は OCR 後に削除する

#### 4.3.4 失敗時の扱い
- 手動入力へ遷移できる
- 再アップロード導線を用意する

#### 4.3.5 未確定データ
- 読み込み後、その場で確定されなかった場合は破棄対象とする
- 画面離脱時に削除できる
- さらに一定時間経過後に自動削除する

#### 4.3.6 確定後に保持する情報
- 店名
- 購入日
- 合計金額
- OCR ステータス
- 作成日時

### 4.4 サブスクリプション管理

#### 4.4.1 登録方法
- 手動登録
- 決済画面スクリーンショットの OCR 登録

#### 4.4.2 OCR フロー
- レシート OCR と同じ基盤を利用する
- OCR 結果を確認、修正してから確定する
- 画像は OCR 後に削除する
- OCR 全文テキストは確認前のみ保持し、確定後に削除する

#### 4.4.3 登録項目
- サービス名
- 金額
- 支払周期: 月額、年額
- 次回支払日
- カテゴリ
- 状態: 有効、停止、解約
- メモ

#### 4.4.4 サブスクカテゴリ
- 初期カテゴリを用意する
- ユーザーごとに追加、編集できる

### 4.5 ダッシュボード・閲覧機能

#### 4.5.1 ダッシュボード
- 今月の支出合計
- 先月比の差額
- 先月比の増減率
- サブスク合計金額
- 直近の支払い予定
- 1 か月カレンダー

#### 4.5.2 一覧表示
- 取引一覧
- サブスク一覧
- 検索、フィルタ、並び替え

#### 4.5.3 カレンダー表示
- 初期表示は月表示
- 日、週、月、年の粒度で切り替えできる
- 支出、収入を切り替えて集計表示できる
- 日ごと、週ごと、月ごと、年ごとの合計を確認できる

### 4.6 通知機能
- アプリ内通知のみを対象とする
- サブスク支払日の事前通知を行う
- ユーザー単位で通知の ON、OFF を設定できる
- ユーザー単位で何日前に通知するかを任意の日数で設定できる

### 4.7 検索・エクスポート

#### 4.7.1 検索
- 取引の店名
- 取引のメモ
- サブスクのサービス名

#### 4.7.2 エクスポート
- 取引 CSV エクスポート
- サブスク CSV エクスポート

## 5. 非機能要件

### 5.1 性能
- 個人利用と少人数デモを想定する
- 同時利用数は 1 から 5 ユーザー程度を想定する

### 5.2 信頼性
- OCR 処理の同時実行数を制限する
- 長時間処理や OCR 失敗が API 全体へ波及しない構成とする
- 未確定 OCR データを定期的に掃除する

### 5.3 セキュリティ
- HTTPS 通信
- パスワードのハッシュ化
- 認証済みユーザーのみデータへアクセスできる
- 定期バックアップを実施する

### 5.4 データ保持
- OCR 用の画像は永続保存しない
- OCR 全文テキストは確認前のみ保持する
- 確定済みの構造化データのみを永続化する

### 5.5 時刻・タイムゾーン
- タイムゾーンはユーザー設定で切り替え可能にする
- 集計、通知、カレンダー表示はユーザー設定タイムゾーンを基準に行う

## 6. システム構成

### 6.1 クライアント
- Next.js App Router による Web アプリ

### 6.2 サーバー
- API: Node.js
- DB: PostgreSQL
- キュー、キャッシュ: Redis
- OCR: Tesseract

### 6.3 運用
- Docker Compose による構成管理
- 自宅サーバー運用

## 7. MVP の対象範囲

### 7.1 含めるもの
- ユーザー認証
- 手動取引登録
- レシート OCR と取引確定
- 手動サブスク登録
- サブスク OCR と確定
- ダッシュボード
- 一覧、カレンダー、検索、CSV エクスポート
- 通知設定とアプリ内通知

### 7.2 含めないもの
- モバイルアプリ本実装
- メール認証
- パスワード再発行
- 外部金融サービス連携
- グループ家計簿
- プッシュ通知

## 8. 対象外
- 商用運用を前提とした大規模スケール
- 銀行、クレジットカードとの自動連携
- 高精度な自動分類 AI
- 家族共有や共同編集

## 9. 今後の拡張
- モバイルアプリ対応
- パスワード再発行とメール認証
- クラウド OCR への差し替え
- クラウドストレージ対応
- 自動分類支援
