# 02. データ設計

## 1. 設計方針
- 個人利用前提のため、すべての主要データは `user_id` に紐づける
- OCR の元画像は永続保存しない
- 確認前の OCR データと、確定後の業務データを分離する
- MVP で確定している項目は JSONB ではなく正規化された列として保持する

## 2. ER 概要

```text
users
  ├─ user_settings
  ├─ transaction_categories
  ├─ subscription_categories
  ├─ receipt_ocr_jobs
  ├─ subscription_ocr_jobs
  ├─ transactions
  │    └─ receipts
  ├─ subscriptions
  ├─ notifications
  └─ refresh_tokens
```

## 3. 主要テーブル

### 3.1 users
- 認証主体
- メールアドレスは一意

| カラム | 型 | 説明 |
| --- | --- | --- |
| id | UUID | 主キー |
| email | VARCHAR(255) | ログイン ID |
| password_hash | VARCHAR(255) | パスワードハッシュ |
| display_name | VARCHAR(100) | 表示名 |
| created_at | TIMESTAMPTZ | 作成日時 |
| updated_at | TIMESTAMPTZ | 更新日時 |

### 3.2 user_settings
- ユーザー単位の表示、通知設定を保持する

| カラム | 型 | 説明 |
| --- | --- | --- |
| user_id | UUID | users への FK |
| timezone | VARCHAR(100) | 例: `Asia/Tokyo` |
| subscription_notification_enabled | BOOLEAN | 通知有効フラグ |
| subscription_notification_days_before | INTEGER | 何日前に通知するか |
| created_at | TIMESTAMPTZ | 作成日時 |
| updated_at | TIMESTAMPTZ | 更新日時 |

### 3.3 transaction_categories
- ユーザーごとの取引カテゴリマスタ
- 未設定は `NULL` で扱う

| カラム | 型 | 説明 |
| --- | --- | --- |
| id | UUID | 主キー |
| user_id | UUID | users への FK |
| name | VARCHAR(100) | カテゴリ名 |
| sort_order | INTEGER | 表示順 |
| is_default | BOOLEAN | 初期カテゴリ由来か |
| created_at | TIMESTAMPTZ | 作成日時 |
| updated_at | TIMESTAMPTZ | 更新日時 |

補足:
- カテゴリ削除時は `transactions.category_id` を `NULL` に設定する（`ON DELETE SET NULL`）
- 削除前にクライアントで「このカテゴリが付いた取引は未分類になります」と確認ダイアログを表示する

初期カテゴリ:
- 食費
- 日用品
- 交通費
- 娯楽
- 通信費
- 光熱費
- 医療費
- 衣服
- 教育
- その他

### 3.5 receipt_ocr_jobs
- レシート OCR の確認前データ
- 確定後または期限切れで削除する

| カラム | 型 | 説明 |
| --- | --- | --- |
| id | UUID | 主キー |
| user_id | UUID | users への FK |
| status | ENUM | `pending`, `processing`, `success`, `failed`, `confirmed`, `expired` |
| extracted_merchant | VARCHAR(255) NULL | 抽出店名 |
| extracted_date | DATE NULL | 抽出日付 |
| extracted_total | DECIMAL(12,2) NULL | 抽出金額 |
| raw_text | TEXT NULL | OCR 全文 |
| expires_at | TIMESTAMPTZ | 自動削除期限 |
| processed_at | TIMESTAMPTZ NULL | OCR 完了日時 |
| confirmed_at | TIMESTAMPTZ NULL | 確定日時 |
| created_at | TIMESTAMPTZ | 作成日時 |
| updated_at | TIMESTAMPTZ | 更新日時 |

補足:
- 元画像パスは保持しない
- 画像ファイル自体は一時領域にのみ存在する

## 4. データ保持ルール

### 4.1 永続化するもの
- ユーザー情報
- 設定
- カテゴリマスタ
- 確定済み取引
- 確定済みサブスク
- 確定済みレシートメタデータ
- 通知

### 4.2 永続化しないもの
- レシート画像
- サブスクスクリーンショット画像

### 4.3 一時保持して後で削除するもの
- 確認前 OCR 全文テキスト
- OCR 確認待ちジョブ

## 5. ライフサイクル設計

### 5.1 レシート OCR
1. ジョブ作成
2. OCR 処理
3. ユーザー確認
4. 確定時に `receipts` と `transactions` を作成
5. `receipt_ocr_jobs.raw_text` を削除
6. ジョブ自体は `confirmed` 後に削除、または監査用に短期間保持後削除

### 5.2 サブスク OCR
1. ジョブ作成
2. OCR 処理
3. ユーザー確認
4. 確定時に `subscriptions` を作成
5. `subscription_ocr_jobs.raw_text` を削除
6. ジョブ自体は期限切れで削除

### 5.3 サブスク次回支払日の自動更新
- 毎日 AM 0:00 に定期ジョブが実行される
- `status = active` かつ `next_payment_date` が今日以前のレコードを対象とする
- `billing_cycle` に応じて `next_payment_date` を加算する
  - `monthly`: +1 ヶ月
  - `yearly`: +1 年
- 複数周期を跨いで滞留していた場合は、今日以降になるまで繰り返し加算する
- `paused`、`cancelled` のサブスクは更新しない
